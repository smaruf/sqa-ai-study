# Level 1 — Basic: Unit Testing Frameworks

Level 1 deepens your understanding of unit testing by exploring the features of each language's primary test framework. You will learn how to organise tests into suites, share setup/teardown logic, use rich assertion libraries, measure code coverage, and write data-driven (parameterised) tests.

---

## Key Concepts

### Test Organisation
- **Test classes / suites** — group logically related tests
- **Setup & Teardown** — code that runs before/after each test (or the whole suite)
- **Fixtures** — reusable objects that tests share without coupling

### Assertions
- Equality, inequality, null checks, type checks
- Exception assertions
- Approximate / floating-point comparisons
- Fluent / matcher-style assertions

### Code Coverage
Coverage measures which lines, branches, and conditions your tests exercise.  
Aim for **meaningful coverage**, not 100% line coverage at all costs.

| Coverage Type | Meaning |
|---------------|---------|
| Line / Statement | Was this line executed? |
| Branch | Was each `if`/`else` arm taken? |
| Condition | Was each boolean sub-expression true *and* false? |

### Parameterised / Data-Driven Tests
Instead of writing 10 nearly identical tests, supply a table of inputs and expected outputs to a single test function.

## How to Run

### Python — with coverage
```bash
pip install pytest pytest-cov
cd level-1-basic/python
pytest -v --cov=. --cov-report=term-missing
```

### Java
```bash
cd level-1-basic/java
mvn test
# Coverage report at: target/site/jacoco/index.html
```

### Golang — with coverage
```bash
cd level-1-basic/golang
go test -v -coverprofile=coverage.out ./...
go tool cover -html=coverage.out -o coverage.html
```

### C\#
```bash
cd level-1-basic/csharp
dotnet test --collect:"XPlat Code Coverage"
# Report generated by coverlet in TestResults/
```

---

## Key Takeaways

1. Use setup/teardown to avoid code duplication, not to hide test dependencies.
2. Parameterised tests dramatically reduce boilerplate while increasing coverage.
3. Coverage is a guide, not a goal — untested *behaviour* matters more than line numbers.
4. Fluent/matcher-style assertions produce better failure messages.
